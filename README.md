## Запуск

В репозитории: 
- `contract`: npm-пакет `@roundsquares/contract` с общими типами и контрактом API
- `server`: Nest.js + PostgreSQL
- `client`: Vite + React 

Пошаговая инструкция:
1. Выполнить `npm install` в `server`, `client`.
2. Выполнить `npm run prepublishOnly` у пакета `contract`.
3. Поднять PostgreSQL, скопировать `server/env.example` в `server/.env` и выставить `DB_URI`, `JWT_SECRET` (`ROUND_DURATION` и `COOLDOWN_DURATION`).
4. Из `server` команда `npm run start:dev` для dev или `npm run build` и `npm start` для production (вход из `dist/main.js`).
5. Из `client` задать `VITE_API_URL` на URL API (`http://localhost:3000`) и выполнить `npm run dev`. 
6. При первичном запуске - таблицы и тестовые учётные записи (`roma:roma` и `admin:admin`) создадутся автоматически.

## Аутентификация

Для аутентификации в этом репозитории используется jwt токен. Технология известная, зарекомендовавшая себя. 
Давайте в ней усомнимся. Как минимум - в целесообразности гонять в каждом http запросе такой большой кусок данных.
Расскажите о проектах с вашим участием, где использовалась другая техника авторизации. 
Если в таких проектах вы бы могли выбирать - выбрали бы вы аутентификацию через jwt? Если да, то почему?
В каком проекте, в котором вы участвовали, вы бы наоборот, вместо jwt выбрали что-то другое и почему?

___

В проектах, где я участвовал, мы использовали JWT для авторизации в REST API (например, в админ панелях или публичных API).
Однако в системах с highload и real-time взаимодействием (игровые серверы, WebSockets) применяли более удобные подходы для этих сфер.

В одном из проектов (онлайн 4000+) мы хранили сессии в Redis, а клиент получал только короткий id сессии.

Это давало:
- Мгновенную инвалидацию - при блокировке пользователя сессия удалялась из Redis, и доступ прекращался сразу, без ожидания истечения JWT.
- Меньший размер передаваемых данных, то есть, в каждом запросе шёл только sessionId, а не токен с клеймами.

Я бы выбрал JWT, если проект представляет собой классический REST API или SPA, где не нужна мгновенная инвалидация и важна stateless-архитектура для горизонтального масштабирования. 

JWT удобен для микросервисов - сервис авторизации не нужен при каждом запросе.
В real-time системах с высокой нагрузкой и требованием к мгновенной блокировке, я предпочёл бы сессии на Redis или аналогичное решение. 
JWT сложно отозвать до истечения его срока жизни, и в некоторых моментах это критично.

## Идентификация

В этом репозтории в некоторых таблицах для идентификации записи используется uuid. Эта техника удобна по нескольким параметрам.
Сложно представить, чтобы использование uuid во всех таблицах - как стандарта - создало бы где нибудь проблему.
Но может быть в вашей практике встречались случаи, где uuid как идентификатор стал проблемой или мог бы стать проблемой?
Опишите где и почему.

___

Использование UUID в качестве первичного ключа в базах данных может создать проблемы.

Основные сложности:
- Фрагментация индексов - случайные значения UUIDv4 нарушают последовательность вставки в B‑tree, что приводит к снижению производительности.
- Размер индекса - UUID занимает больше, чем BIGINT. В таблицах с миллионами строк, разница в размере индекса сильно существенна, что замедляет, например, операции с JOIN.
- Кэширование - в InnoDB (MariaDB/MySQL) меньший размер ключа позволяет уместить больше записей в буф. пуле.

Как я подходил к этому в своих проектах:
При проектировании схемы данных я использовал двойной подход:
- Внутренние связи (или внешние ключи) - BIGINT AUTO_INCREMENT, для эффективности JOIN и компактности индексов.
- Публичные идентификаторы для API - UUID для скрытия количества записей и безопасной передачи на клиент.

Такой подход позволял сохранить производительность базы данных на сильно загруженных проектах и одновременно получить удобство UUID на уровне хорошего приложения.

## Организация кода

В этом репозитории, серверный код сделан на базе NestJS. 
DI, services, controllers, middleware, decorators - в этом, как и во многих других подходах есть плюсы и выгоды.
Расскажите с примерами из опыта о реальной пользе того или иного механизма организации кода лично для вас или
для вашей команды, а может быть - для компании. Если есть какой-то механим, который также - для вас, команды 
или компании наоборот - создавал сложности - тоже расскажите.

___

В проектах, где я использовал Nest.js, его механизмы организации кода приносили ощутимую пользу.

Польза Dependency Injection (DI):
- Тесты. Благодаря DI, я легко заменял реальные сервисы на моки при написании unit-тестов. На мой взгляд, это очень ускоряло разработку и сильно повышало надёжность.
- Модульность. Разбивал функциональность на модули. Каждый модуль чётко определял, что он предоставляет наружу, а что скрывает внутри. 
В целом, это сильно упрощало разделение работы в команде (возможность вести свои модули без конфликтов).
- Переиспользование. Сервисы легко инжектились в несколько контроллеров или других сервисов без необходимости создавать экземпляры самому.

Сложности, с которыми сталкивался:
- Избыточность для небольших проектов. Для простых сервисов настройка модулей и провайдеров излишна по сравнению с экспресс‑подходом.
- Сложность отладки циклических зависимостей. Когда два модуля зависели друг от друга, Nest.js выдавал не всегда очевидные ошибки, и поиск места циклической ссылки требовал времени.

Для крупных проектов, я бы предпочёл Nest.js с его DI и строгой структурой.

## Реактивность

В этом репозитории, для интерфейса используется React. Его название связано с реактивным подходом к отрисовке
интерфейса - мы меняем состояние, а интерфейс перерисовывается в соответствии с этим. 
Сам посыл подкупает - разработчик заботится о состоянии, остальное за него делает фреймворк. 
Такой подход, конечно, помимо React реализуют и многие другие фреймворки. 
Были ли ситуации в вашем опыте, когда реактивность была не плюсом, а помехой? Если да, расскажите на примере.
Если был опыт построения рендера интерфеса на другом принципе, раскажите на каком и какие были резльтаты?
Расскажите на конкретных примерах, что вам доставляет неудобства в том, как реактивность реализована
непосредственно во фреймворке React?

___

В моей практике реактивность чаще всего была плюсом, но были ситуации, когда в ней не было необходимости.

Когда реактивность мешала:
- Бесполезные перерисовки. В одном проекте с клиентом на Vue 3 (сложный UI с большим количеством динамических компонентов), изменение одного свойства в store вызывало перерисовку слишком большого количества компонентов. 
Приходилось вручную оптимизировать с помощью computed и shallowRef, чтобы поднять производительность.
- Сложность отладки. В компонентах с глубокой вложенностью, реактивных данных бывает сложно понять, почему компонент перерисовывается (или наоборот). Для меня, это усложняло дальнейшую поддержку.

Опыт с другим принципом рендеринга:
- В некоторых проектах (например, на чистом JS с прямым управлением DOM), был полный контроль над обновлениями, но нужно было самому отслеживать, какой элемент и когда обновлять.
Реактивные фреймворки (например Vue или React) значительно ускоряют фронтенд разработку, даже с их недостатками.

Неудобства в реализации реактивности (на примере Vue):
- Механизм watch иногда вводил в заблуждение, особенно при наблюдении за вложенными объектами.

При работе с reactive, иногда терялась реактивность - это постоянная ошибка, требующая внимания.

В целом, для меня, реактивность - супер мощный инструмент, но требует полноценного понимания внутренних механизмов, чтобы в дальнейшем, не было проблем с производительностью, поддержкой и масштабированием.
